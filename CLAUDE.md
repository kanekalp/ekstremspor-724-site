```sql
create table equipments (
  id           uuid primary key default gen_random_uuid(),
  type         text not null check (type in ('bicycle','skates','skateboard')),
  status       text not null default 'available'
                 check (status in ('available','in_use','damaged')),
```

## ROW LEVEL SECURITY

Enable RLS on every public table. Use the **consolidated pattern**: one
policy per (table, action) so PostgREST does not OR together multiple
permissive policies (lint 0006). Wrap every `auth.uid()` and
`private.is_admin()` call in `(select …)` so Postgres caches the value
once per query instead of re-evaluating per row (lint 0003).

The admin-check function lives in the `private` schema — it is NOT in
`api.schemas`, so PostgREST does not expose it via RPC, which silences
lint 0028/0029 while RLS continues to call it directly.

```sql
-- Admin check, hidden from REST
create schema if not exists private;
grant usage on schema private to anon, authenticated;

create or replace function private.is_admin()
  returns boolean
  language sql security definer stable
  set search_path = public
as $$
  select exists (
    select 1 from public.profiles
    where id = auth.uid() and role = 'admin'
  );
$$;
grant execute on function private.is_admin() to anon, authenticated;

-- ─── profiles ───────────────────────────────────────────────
alter table profiles enable row level security;

create policy "profiles_select" on profiles for select using (
  (select auth.uid()) = id
  or (select private.is_admin())
  or exists (
    select 1 from activities a
    where a.user_id = profiles.id and a.status = 'approved'
  )
);
create policy "profiles_insert" on profiles for insert
  with check ((select auth.uid()) = id);
create policy "profiles_update" on profiles for update
  using ((select auth.uid()) = id or (select private.is_admin()));

-- ─── activities ─────────────────────────────────────────────
alter table activities enable row level security;

create policy "activities_select" on activities for select using (
  status = 'approved'
  or (select auth.uid()) = user_id
  or (select private.is_admin())
);
create policy "activities_insert" on activities for insert with check (
  ((select auth.uid()) = user_id and source = 'remote')
  or (source = 'on_site' and (select private.is_admin()))
);
create policy "activities_update" on activities for update
  using ((select private.is_admin()));
create policy "activities_delete" on activities for delete using (
  ((select auth.uid()) = user_id and status = 'pending')
  or (select private.is_admin())
);

-- ─── equipments ─────────────────────────────────────────────
alter table equipments enable row level security;

create policy "equipments_select" on equipments for select using (true);
create policy "equipments_insert" on equipments for insert
  with check ((select private.is_admin()));
create policy "equipments_update" on equipments for update
  using ((select private.is_admin()));
create policy "equipments_delete" on equipments for delete
  using (status <> 'in_use' and (select private.is_admin()));

-- ─── event_config ───────────────────────────────────────────
alter table event_config enable row level security;

create policy "event_config_select" on event_config for select using (true);
create policy "event_config_update" on event_config for update
  using ((select private.is_admin()));
```

**Rules:**
- Never write a second permissive policy for the same (table, action) —
  fold the condition into the existing one with OR.
- Never call `auth.uid()` or `private.is_admin()` bare in a policy —
  always `(select …)`.
- New admin-only logic in RLS goes through `private.is_admin()` so the
  function never re-enters the `public` API surface.

## MIDDLEWARE — ROUTE GUARD

`middleware.ts` (project root):

```typescript
import { createServerClient } from "@supabase/ssr";
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";

export async function middleware(request: NextRequest) {
  const response = NextResponse.next();

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        get: (name) => request.cookies.get(name)?.value,
        set: (name, value, options) => {
          response.cookies.set({ name, value, ...options });
        },
        remove: (name, options) => {
          response.cookies.set({ name, value: "", ...options });
        },
      },
    },
  );

  const {
    data: { user },
  } = await supabase.auth.getUser();

  if (request.nextUrl.pathname.startsWith("/admin")) {
    if (!user) {
      return NextResponse.redirect(new URL("/", request.url));
    }
    const { data: profile } = await supabase
      .from("profiles")
      .select("role")
      .eq("id", user.id)
      .single();

    if (profile?.role !== "admin") {
      return NextResponse.redirect(new URL("/", request.url));
    }
  }

  return response;
}

export const config = {
  matcher: ["/admin/:path*"],
};
```

## AUTH FLOW

> Magic-link email delivery and OTP verification are removed.
> Users sign in with email only — no OTP, no inbox required.
> The server generates a magic-link token internally (no email sent)
> and verifies it client-side to establish the session.

### Sign-In Flow

```
User types email → AuthModal calls signInUser() server action
→ Server creates user if new (email_confirm: true, service role)
→ Server calls admin.generateLink({ type: 'magiclink' }) — no email sent
→ Returns hashed_token to client
→ Client calls supabase.auth.verifyOtp({ token_hash, type: 'email' })
→ Session cookie set → page reloads → user authenticated
```

### Server Action — `lib/actions/auth.ts`

```typescript
"use server";
import { createAdminClient } from "@/lib/supabase/admin";

export async function signInUser(email: string) {
  const admin = createAdminClient(); // SUPABASE_SERVICE_ROLE_KEY

  // Create user if not exists (email_confirm:true skips verification)
  await admin.auth.admin.createUser({ email, email_confirm: true });

  // Generate token server-side — no email is sent
  const { data, error } = await admin.auth.admin.generateLink({
    type: "magiclink",
    email,
  });

  if (error || !data?.properties?.hashed_token) {
    return { error: "Oturum oluşturulamadı." };
  }

  return { tokenHash: data.properties.hashed_token };
}
```

## USER FLOWS

### Flow A — On-Site Entry (admin only)

```
Admin panel → "Stant Girişi" modal
→ Search user by email or name (ILIKE query on profiles)
→ Enter KM + vehicle type (bicycle / skates / skateboard / running)
→ Insert: source = 'on_site', status = 'approved'
→ Leaderboard updates in real time
```

Running is a valid vehicle for on-site entry — admin records it the
same way as any other ride. No equipment is consumed for running.

### Flow B — Remote Entry (user)

```
Landing page → "Sürüş Ekle" button (visible when logged in)
→ Modal:
    Step 1: Choose mode — "Stanttayım" / "Uzaktan"
    Step 2: Vehicle type, plus KM + date range if remote
            (Running is offered only in Uzaktan flow)
    Step 3 (remote only): Screenshot upload + KVKK consent checkbox
→ on_site insert: vehicle ∈ {bicycle,skates,skateboard}, distance=0,
                  source='on_site', status='pending'
                  (Admin assigns equipment, then enters KM on return,
                  which flips status to 'approved'.)
→ remote insert: source='remote', status='pending'
→ Visible in user's "Aktivitelerim" as Pending until approved
→ Does NOT appear on leaderboard until approved
```

### Flow C — Equipment Request Lifecycle

```
User onboarding picks equipment_need ∈ {bicycle, skates, skateboard, none}
→ if not 'none', profiles.equipment_request_status = 'pending'
→ Admin sees request in "Ekipman Talepleri" tab
  → Assign: picks an available unit of that type → status='in_use',
            assigned_to=user, request_status='fulfilled'
  → Reject: request_status='rejected' (user sees this in their modal)
→ User can cancel their own pending request: equipment_need='none',
  request_status='not_needed'
```

### Flow D — Equipment Lifecycle (admin only)

```
add        : Admin creates equipment (auto code 'BSK-001'/'PTN-001'/'KYK-001'
             or custom). status = 'available'.
available → in_use : Admin clicks "Ata"
  → Search and select user, or assign via on_site pending request
  → Set assigned_to, assigned_at
  → status = 'in_use'

in_use → available : Admin clicks "İade Al"
  → Enter KM the user did (approves their pending on_site activity)
  → Optional: mark as 'damaged' if equipment returned damaged
  → Clear assigned_to, set returned_at

available ↔ damaged : Admin can mark/unmark damaged equipment
                      (damaged units are excluded from assignment)

delete     : Admin can delete equipment that is NOT in_use

Equipment is reusable. Historical log preserved via timestamps.
```

### LAN testing (network-agnostic)

`npm run dev` binds to `0.0.0.0` (`next dev -H 0.0.0.0`) so any device
on the same LAN can reach the Next.js server. Find the dev box's
current LAN IP (e.g. `ipconfig` on Windows / `ifconfig`/`ip a` on
*nix) and open `http://<that-ip>:3000` from the phone or tablet.

`NEXT_PUBLIC_SUPABASE_URL` in `.env.local` is kept as
`http://127.0.0.1:54321` — that is only used by server-side code on the
dev box. The browser client (`lib/supabase/client.ts`) rewrites the
hostname to `window.location.hostname` at runtime, so Supabase is
reached at `http://<that-ip>:54321` from each visiting device with no
per-network env edit. This works because the Supabase CLI exposes its
ports on all interfaces by default.

## REALTIME PUBLICATION

Supabase's `supabase_realtime` publication must explicitly include
every table the UI subscribes to. Without it, `.on('postgres_changes')`
subscriptions are silent and the UI only refreshes on full reload.

Tables that MUST be in the publication:
`profiles`, `equipments`, `activities`, `event_config`

The `20260516000001_running_realtime_forest.sql` migration adds them
defensively. When adding a new table, append it to that publication.

In addition to realtime, every admin write path (assign, reject,
reopen, approve, return) should call `refetch()` immediately after
the write resolves. Realtime is fast but not instant; the explicit
refetch removes any perceived UI lag and protects against publication
misconfiguration.

## THEME LOAD STRATEGY

The hero uses CSS custom properties driven by `data-theme` on `<html>`.
To prevent a flash from the `:root` default (morning) to the user's
actual hour-of-day on first paint, an inline pre-paint script in
`app/layout.tsx` computes the theme synchronously before any HTML
parses the body. `ThemeApplier` only handles in-session transitions
(hour boundary while the page is open) — CSS transitions on the custom
properties animate those changes smoothly over ~2 s.

Rule: render with the current hour's theme on load (no animation),
animate on subsequent transitions. Never reset the transition CSS
imperatively; that's what caused the previous flash.

## RUNNING ACTIVITY

Running is a fourth vehicle_type but has NO equipment representation.

- `VehicleType` = `'bicycle' | 'skates' | 'skateboard' | 'running'`
- `EquipmentVehicleType` = `VehicleType` minus `'running'`
- `equipments.type` CHECK constraint excludes running
- `activities.vehicle_type` CHECK constraint includes running

UI inclusion rules:
- AddActivityModal: shown only in 'Uzaktan' mode (no on-site equipment)
- OnSiteEntryModal (admin): shown — admin enters km directly, approved
- Leaderboard filter: shown
- OnboardingModal `equipment_need`: NOT shown (use 'none' for runners)
- EquipmentBand / EquipmentTable / EquipmentRequests: NOT shown

## FOREST NAMING MILESTONE

When `total_km >= target_km` (default 5000), `SaplingProgress` flips
into a celebratory state:
- Card border/ring shifts to grass-deep
- Header copy switches to "Kampüsümüzün ormanı hazır"
- A banner appears suggesting the community vote on the forest name

The chosen name is stored in `event_config.forest_name` (nullable).
Stand team coordinates the actual vote/decision; the app surfaces the
milestone but does not implement the voting UI in scope.
